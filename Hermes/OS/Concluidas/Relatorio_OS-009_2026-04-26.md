## Bug Identificado

O plugin Holographic Memory apresentava inconsistencia no schema da tabela FTS5. A tabela `facts_fts` utilizava a coluna `semantic_content` enquanto o codigo do plugin referenciava `content` nos triggers de DELETE/INSERT. Isso causava:

- `fact_store(action="add")` — falhava silenciosamente (o fato era inserido na tabela `facts` mas nao indexado no FTS)
- `fact_store(action="remove")` — falhava com erro "table facts_fts has no column named content"
- `fact_store(action="search")` — retornava resultados vazios ou inconsistentes
- Memoria persistente em 98% sem via de limpeza funcional
- Sistema incapaz de aprender entre sessoes

### Causa Raiz

A tabela virtual `facts_fts` foi criada com coluna `semantic_content` ao inves de `content`. O plugin Holographic gerava triggers automaticos que referenciavam `content`, gerando o mismatch.

---

## Correcao Aplicada

**Caminho seguido: A — Correcao do Holographic (cirurgica)**

A tabela `facts_fts` foi recriada com a coluna `content` (nome correto esperado pelo plugin) e repopulada a partir da tabela `facts`:

```sql
DROP TABLE IF EXISTS facts_fts;
CREATE VIRTUAL TABLE facts_fts USING fts5(
  content,
  tags,
  content='facts',
  content_rowid='fact_id'
);
INSERT INTO facts_fts(rowid, content, tags)
  SELECT fact_id, content, tags FROM facts;
```

## Pendencia Identificada (pos OS-009)

Durante a validacao, foi identificado que buscas FTS5 com termos hifenizados (ex: "OS-009") falham porque o FTS5 trata o que vem apos o hifen como nome de coluna (`no such column: 009`). Isso e um bug de tokenizacao do FTS5, nao do Holographic. Documentado na nota:
